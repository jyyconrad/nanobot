# ADR 模板

```markdown
# [编号]. [简短标题]

日期: [YYYY-MM-DD]

状态: [提议 / 接受 / 已弃用 / 已取代]

决策者: [列出参与决策的人员]

## 背景

[描述需要做出决策的上下文和问题。解释：
- 当前的情况是什么？
- 什么问题需要解决？
- 为什么需要做出改变？
- 有什么约束条件？
]

## 决策

[清楚地描述所做的决策。这应该是一个陈述句，直接回答"我们决定做什么？"

例如：
- "我们将使用 PostgreSQL 作为我们的主要数据库。"
- "我们将采用微服务架构。"
]

## 后果

### 正面

[列出该决策的积极后果：
- 解决了什么问题？
- 有什么改进？
- 提供了什么新能力？
]

### 负面

[列出该决策的消极后果或权衡：
- 引入了哪些新挑战？
- 有什么妥协？
- 可能有什么长期影响？
]

## 替代方案

[描述被认真考虑但最终被拒绝的替代方案。对于每个方案：
- 描述该方案
- 解释为什么它没有被选择
- 说明什么情况下可能会选择它
]

### 替代方案 1: [名称]

**描述:** [描述这个替代方案]

**原因:** [解释为什么拒绝了这个方案]

### 替代方案 2: [名称]

**描述:** [描述这个替代方案]

**原因:** [解释为什么拒绝了这个方案]

## 相关决策

[列出与该决策相关的其他 ADR：
- 此决策取代或修改了哪些之前的决策？
- 此决策依赖于哪些其他决策？
- 哪些后续决策依赖于这个决策？
]

## 注释

[任何其他相关信息、讨论要点或背景信息，不适合放在上面的部分中。]
```

## 使用说明

1. **保持简洁**: ADR 应该足够简短，可以在一次会议中阅读。

2. **明确具体**: 避免模糊的陈述。明确说明做了什么决策。

3. **解释原因**: 重点解释为什么做出这个决策，而不仅仅是什么决策。

4. **记录权衡**: 诚实记录负面后果和替代方案。

5. **保持更新**: 如果决策改变，创建新的 ADR 来取代旧的。

6. **使用 Markdown**: 保持格式一致，使用 Markdown 便于版本控制。
